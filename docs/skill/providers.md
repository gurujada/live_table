# Providers And Context-Owned Queries

Use this when a LiveTable's base query needs joins, selected maps, computed fields, parent ids, category ids, tenant ids, current user, seat type, feature mode, or other context-owned arguments.

## Provider With Arguments

Assign a provider tuple in `mount/3`:

```elixir
def mount(%{"program_id" => id}, _session, socket) do
  {:ok, assign(socket, :data_provider, {MyApp.Programs, :list_colleges, [id, :open]})}
end
```

The provider owns the base query:

```elixir
def list_colleges(program_id, seat_type \\ :open) do
  from p in Program,
    join: cp in CollegeProgram,
    on: cp.program_id == p.id,
    join: c in College,
    as: :college,
    on: c.id == cp.college_id,
    join: rc in RankCutoff,
    as: :rank_cutoff,
    on: rc.college_program_id == cp.id and rc.seat_type == ^seat_type,
    where: p.id == ^program_id,
    select: %{
      program: p,
      college: c,
      opening_rank: rc.opening_rank,
      closing_rank: rc.closing_rank,
      seat_type: rc.seat_type
    }
end
```

## Prefer Stable Provider State

When a UI control changes provider arguments, first ask whether it is really a filter or sort. If it is, use a `Transformer` and keep the provider stable.

Use provider arguments for true base-query inputs:

- parent id
- tenant/current-user scoping
- feature mode that changes the base dataset
- route-owned identity such as program id or organization id

Use transformers for query variants:

- rank or eligibility filter
- top-N limit
- custom sort mode
- date-window filtering
- domain-specific query branch

## Changing Provider Arguments

For a true provider-argument change, update `:data_provider` and push a patch so LiveTable's generated flow reloads with the new provider already assigned:

```elixir
def handle_event("change_seat_type", %{"seat_type" => seat_type}, socket) do
  seat_type = String.to_existing_atom(seat_type)

  provider =
    {MyApp.Programs, :list_colleges, [socket.assigns.program_id, seat_type]}

  socket =
    socket
    |> assign(:data_provider, provider)
    |> assign(:seat_type, seat_type)
    |> push_patch(to: ~p"/programs/#{socket.assigns.program_id}")

  {:noreply, socket}
end
```

Avoid overriding LiveTable's generated `handle_params/3` unless the library version explicitly supports that extension point. For many cases, model the control as a `Transformer` instead of changing the provider.

## Avoid Manual Refresh Helpers

Do not copy code that calls:

```elixir
stream_resources(fields(), options, data_provider)
```

That helper is generated by `LiveTable.LiveResource` and is not the application authoring API. Existing direct usage should be treated as legacy/app-specific code.
